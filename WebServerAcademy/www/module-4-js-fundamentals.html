Objects

A JavaScript object can be looked at as a simple collection of Properties

Objects implement a Map Data Structure, where properties are containers that hold primitives or other objects 

                                    
// object literal syntax makes creating objects extremely easy
var obj = { a: 42, b: true, c: { } };

obj.a; // 42
obj.b; // true
obj['c']; // {}

var key = 'd';
obj[key] = 'AC';
obj.d; // 'AC';

Everything that is not a primitive is an Object

This includes Functions and Arrays!
  • Object literal syntax makes it extremely easy to create objects on the fly
  • Latter own we will see other ways we can use to create more complex objects in javascript
next: access to object properties can somehow be configured through attributes

Property Access

Objects can have properties read or written to, as well as new ones added or removed dynamically 

                                    
// 1. Dot syntax
newObject.someKey = "Hello"; // create a new property if non existent and update value
var value = newObject.someKey; // get property value

// 2. Square bracket syntax
newObject["someKey"] = "Hello"; //  create a new property if non existent and update value
var value = newObject["someKey"]; // get properties

// 3. Object.defineProperty (ES5 only)
// create a new property if non existent and update attributes and value
Object.defineProperty( newObject, "someKey", {
    value: "for more control of the property's behavior",
    writable: true,
    enumerable: true,
    configurable: true
});

// 4. delete property
delete newObject.someKey;
  • In ES5, each property has attributes that determines how they can be used
  • messing with property attributes is not a common thing to do, in js we embrace the default publicly accessible model of object properties and use other techniques to restrict access to data

References

Variables can hold the actual values of primitives types or references to objects 

                                    
var a = 80; // a holds the value of 80
var b = a; // b holds the value of 80

var a = { n: 80 }; // a holds a reference to the object
var b = a; // b holds a reference to the same object

console.log(a.n); // 80
console.log(b.n); // 80
a.n = 100;
console.log(a.n); // 100
console.log(b.n); // 100
next: js typing system is dynamic, lets see what this means

Dynamic Typing

The type of a variable does not have to be declared and can be reassigned to a different type 

                                    
var bootcamp = 10; // bootcamp is a Number
bootcamp = 'Lisbon'; // bootcamp is now a String
bootcamp = true; // bootcamp is now a Boolean
bootcamp = { edition: 10, city: 'Lisbon', running: true }; // bootcamp is now an Object

typeof operator returns a string indicating the type of the value 

                                    
console.log(typeof bootcamp); // prints object to the console
  • In a static type system, we guarantee at compile time that all program executions will be free of type errors
  • In a dynamic type system, we wait until runtime and throw an exception if a type error is detected
  • Dynamic Typing is a double-edged sword.. It provides lots of flexibility, but sometimes at great cost...
  • A static type system allows for much better development tools, and is less bug prone

In JavaScript, functions are ordinary objects with the

  • A function that returns nothing is equivalent to returning undefined 
                                        
function foo(a) {

Primitive Wrappers

Primitives are immutable and have no methods, but JavaScript has the Boolean, Number and String objects that wrap around primitive values 

                                    
var str = 'this is a string primitive';
typeof str; // 'string'

// due to implicit coercion of string to a wrapper object,
// we can invoke methods on it
console.log(str.toUpperCase());

// the new operator is the way to instantiate wrapper objects
var strObject = new String(str);
typeof strObject; // 'object'

var strPrimitive = strObject.valueOf();
typeof strPrimitive; // 'string'

Wrapper objects are immediately destroyed after their use!

next: JS has what is known as lexical scope (also known as static)

Lexical Scope

JavaScript has support for static lexical scope only
Variable scope depends on its position on the source code 

                                        
// a global definition of animal
var animal = 'Horse, ';

function mammal() {

    // a local animal variable
    var animal = 'Dog, ';

}

mammal();
animal += ' a mammal with four legs';

// Dog or Horse?
console.log(animal);

Variables in JavaScript can have either Global or Local scope

  • lexical - to do with words, also known as static scope, as the scope depends on the location of the variable on the source code
  • on dynamic scopes, the scope does not depend on where a function was declared, but where it was called from
  • javascript does not have dynamic scope, although the binding of the this variable closely relates the dynamic scope mechanism

Local Scope

Local variables exist only within the function body
of which they are defined 

                                    
console.log(dog); // bad things will happen

function animal() {

    var dog = 'Dob'; // accessible inside animal function only

}
Please note that contrary to other programming languages, ES5 local scope is bound to function and not block!
  • the block scope causes some interesting behaviors, to be explored in a few slides

Global Scope

Global variables live through the entire execution of the runtime
and can be accessed and altered in any scope 

                                    
var dog = 'Dog'; // global definition of dog

function animals() {

    console.log(dog); // access to global dog
    badAnimal();
    console.log(rabbit); // access to global rabbit
}

function badAnimal() {

    rabbit = 'rabbit'; // rabbit leaked into the global scope
}

Variables defined outside a function or without the var keyword will be bound to the Global Scope

ES5 contains a strict mode, causing the interpreter to check and scream at evil things such as implicit globals
  • It is generally a bad idea to run without strict mode on, unless you want to run code written a decade ago..

next: function scope introduces an interesting behavior called hoisting

Hoisting

Variable declarations are always processed before any code is executed 

                                        
// global animal variable
var animal = 'dog';

function showAnimal() {

    if (animal != 'cat') {

        console.log(animal);

        // declare and assign variable
        var animal = 'cat';

        console.log(animal);
    }
}

// can you guess the output?
showAnimal();
                                        
                                    
                                        
// global animal variable
var animal = 'dog';

function showAnimal() {

    var animal; // declare variable
    if (animal != 'cat') {

        console.log(animal);
        animal = 'cat'; // assign variable
        console.log(animal);
    }
}

// what about now?
showAnimal();
This behavior is called hoisting, as it appears that the declaration is moved to the top of the scope
next: we mentioned before that functions were simply js objects that could be passed around, which accounts for great power and expressiveness in JavaScript. We say they are first class citizens in JS.

High Order Functions

Functions that operate on other functions, either by taking them as arguments or by returning them, are called higher-order functions 

                                    
// build a multiplier by 2 and store it
// in the variable named double
var double = multiplier(2);

// use it to double some numbers
console.log(double(5)); // 10
console.log(double(3)); // 6

// builds a new multiplier
function multiplier(operand1) {
    return function(operand2) {
        return operand1 * operand2;
    }
}

Remember that functions in javascript are just ordinary objects

next: arrays allow for a very elegant functional style of programming

Functional Style with Arrays

The functional style can make code much cleaner and easier to maintain, although some performance penalty is paid 

                                    
// apply a filter function to an array, obtaining the older persons
var persons = [{ name: 'Rui', age: 43 }, { name: 'Pedro', age: 29 }];
var old = persons.filter(function(person) {
    return person.age > 40;
});

// apply a transform function to an array, transforming each element in the array into its square root
var numbers = [1, 4, 9];
var roots = numbers.map(Math.sqrt); // [1, 2, 3]

// reduce an array to one single value, the total number of characters in the array
var animals = ['cat', 'dog', 'fish'];
var letters = animals.reduce(function(sum, word) {
    return sum + word.length;
}, 0); // 10

// most array functions can be chained together, this example flattens an array and finds even numbers
const data = [[1,2,3], [4,5,6], [7,8,9]];
const evens = data.reduce(function(total, value) {
    return total.concat(value);
}, []).filter(function(value) {
    return value % 2 === 0;
});
The original array objects are not mutated by map, filter or reduce
  • Functional programming is a declarative programming style where all functions have no side effects and data is readonly
  • You can explain this through the definition of a mathematical function
  • If something needs to be changes, create a new instance instead
  • Benefits include: simpler code, less bugs, concurrency safety
  • Downsides include: higher CPU, memory
