Composition

Design technique to implement HAS-A relationship in classes

  • The act of composing a class out of references to other objects;
  • With composition, references to the constituent objects become fields of the containing object.

Inheritance

Inheritance is the design technique to implement IS-A relationship in classes. It allows classes to be derived from other classes, thereby inheriting their fields and methods.

  • For example, we belong to the mammals category. But so do dogs and cats.
  • This means that all of us share same properties and behaviours of the mammal class but we also have our own properties and behaviours which set us apart from each other.

Why inheritance?

Allows you to reuse the fields and methods of the existing class without having to write them again (easier to use and modify).

  • Well because using inheritance in programming, mostly prevents you from repeating code and also provides more scalable solutions.
  • Let’s think about the calculator. On the last example, the calculator had a brand and a color and was able to add numbers.
  • If we think about a basic calculator it can also subtract, divide and multiply numbers.

The Scientific Calculator IS-A Calculator.

  • But now imagine we have a scientific calculator.
  • In addition to everything the basic calculator does it also performs exponents, logarithms, trigonometry functions and so on.
  • Both calculators share the same basic functionality.

The Scientific Calculator can do everything a Calculator does and more.

  • So basically, the scientific calculator can inherit everything from the basic one and just add its specific behaviours.
  • In Java we use the extends keyword to indicate that a class inherits all properties and behaviours from another more general class.
  • The more general class is usually called superclass or parent class, while the one extending is usually called the subclass or child class.
  • So the advantage of using inheritance is that we are sharing the same base class to provide equal functionality to more specific classes by reusing the code.
  • We implement the shared functionality on the superclass and on each subclass we just add the specific properties and behaviours.
  • And moreover If we need to modify or add new behaviours or properties to all our child classes we just need to modify the parent class, instead of changing each class individually.

Use Inheritance when

  • The subclass is a more specific type of the superclass;
  • Share behaviours among classes of the same general type;
  • The subclass and the superclass pass the IS-A test.

Do not use Inheritance

  • Reuse code from another class without existing a proper relationship among the classes;
  • The subclass and the superclass do not pass the IS-A test;
  • The relationship is better described using the HAS-A test.

Inheritance vs Composition

Composition and delegation can be used as an alternative to inheritance.

Inheritance is a good strategy when a close relationship exists in between parent and child object

Inheritance couples objects very close

Composition and Delegation is the more flexible way to express a relationship between classes

Method Overriding

Declaring a method in a subclass which is already present in a parent class is known as method overriding 


/**
 * ScientificCalculator.java
 */

// The Scientific Calculator IS-A Calculator
public class ScientificCalculator extends Calculator {

    @Override
    public void powerOn() {

        super.powerOn(); // call the power method on the superclass
        display.showMessage("And I can also do some more advanced operations such as powers");
    }

}

Default Constructor

The Java compiler will insert a default constructor on a class
if no constructor is found 


// Calculator Class
public class Calculator {

    // default constructor contains no arguments and an empty body
    public Calculator() {}

}
                    

// ScientificCalculator class
public class ScientificCalculator extends Calculator {

    // default constructor includes a call to super if inheriting
    public ScientificCalculator() {
        super();
    }

}

Non-default Constructor

The superclass default constructor is always implicitly called from any subclass constructor 


public class ScientificCalculator extends Calculator {

    private String brand;

    public ScientificCalculator() {
        super(); // implicict constructor call will be added by compiler if not present
        this.brand = brand;
    }
}

No Default Constructor

What if the super class has no default constructor? 


public class Calculator {

    private String brand;

    public Calculator(String brand) {
        this.brand = brand;
    }

}

public class ScientificCalculator extends Calculator {}

Program will not compile, default subclass constructor calls non existing super class default constructor 


public class ScientificCalculator extends Calculator {

    public ScientificCalculator(String brand) {
        super(brand); // superclass has no default constructor, explicit call to super is required
    }

}

Magic Lamp

  • Releases a new genie every time it is rubbed
  • When enchanted (instantiated), the maximum number of genies is set
  • Genies can be Friendly (even) or Grumpy (odd)
  • When the number of genies is exhausted, it releases a recyclable demon
  • Has the ability to recharge itself by recycling a demon
  • We can compare two lamps by their capacity, number of remaining genies and number of times it has been recharged

Friendly Genie

  • When released from the lamp (instantiated), the maximum number of wishes is set
  • It can only grant one wish at a time
  • It can grant wishes if the maximum has not been reached

Grumpy Genie

  • When released from the lamp (instantiated), the maximum number of wishes is set
  • It can only grant one wish at a time
  • Grants ONLY ONE wish, even if the maximum has not been reached

Recyclable Demon

  • When released from the lamp (instantiated), the maximum number of wishes is set
  • It can only grant one wish at a time
  • Grants ALL wishes, even if the maximum has already been reached
  • It can recharge a magic lamp, if recycled
  • It will not grant any more wishes after being recycled
  • It can only be recycled once

Polymorphism

The ability to use superclass variables to refer to subclass objects, and using overridden methods of the subclass even though the subclass isn’t known when the program is compiled.

Polymorphism

Let's create a collection of Calculators


/**
 * Main.java
 */
public class Main {

    public static void main(String[] args) {

        Calculator calculator = new Calculator("Casio", "White");
        Calculator calculatorChinese = new Calculator("Chinese", "Red");

        ScientificCalculator calculatorSci = new ScientificCalculator("Texas Instruments", "Black");
    }
}

Let's group all Calculators


/**
 * Main.java
 */

public class Main {

    public static void main(String[] args) {

        Calculator calculator = new Calculator("Casio", "White");
        Calculator calculatorChinese = new Calculator("Chinese", "Red");

        ScientificCalculator calculatorSci = new ScientificCalculator("Texas Instruments", "Black");

        // we can store all calculators on an array of the superclass type
        Calculator[] myCalculators = { calculator, calculatorChinese, calculatorSci };
    }
}

The reference type can be the superclass of the actual object.

Give them some power


/**
 * Main.java
 */

public class Main {

    public static void main(String[] args) {

        // ...

        // we can store all calculators on an array of the superclass type
        Calculator[] myCalculators = { calculator, calculatorChinese, calculatorSci };

        for(int i = 0; i < myCalculators.length; i++) {

            // the method on the right class (super or sub) will be called at runtime
            myCalculators[i].powerOn();
        }
    }
}

At runtime, the correct method will be called according to the object instance (super or subclass).

Why Polymorphism?

  • Write code that will work without knowing specifically which subclass instance will be chosen
  • Possibility to add more subclasses without breaking the existing code

Abstract Classes

Conceptual classes that cannot be instantiated, but can be subclassed

Why Abstract Classes?

Providing a base skeleton with properties and methods with the sole purpose of being extended

Character Class


/**
 * Character.java
 */

public abstract class Character {

    private String name;

    public abstract void talk(String message);

}

There is no point in creating Character object instances.
The class must be extended.

Abstract methods

Methods that ensure the existence of a specific behaviour without declaring an implementation

A class with an abstract method must be abstract as well

Declaring an Abstract method


/**
 * Character.java
 */

public abstract class Character {

    private String name;

    // I want my characters to talk but I have no idea how each one will do it
    public abstract void talk(String message);

}

An abstract method has no body.
The method must be overriden.

Overriding the Abstract method


/**
 * Yoda.java
 */

public class Yoda extends Character {

    @Override
    public void talk(String message) {
        System.out.println(message);
    }
}

A subclass must implement all the abstract methods,
otherwise it must be abstract as well.

The Object Class

  • Every class, in Java, extends class Object;
  • When you don't explicitly extend another class, you are implicitly extending the Object class.
